# Техническое задание: сайт сети ресторанов «Чашка кофе»

## 1. Назначение

Документ предназначен для реализации нового сайта через Codex на базе репозитория `di-sukharev/vibe`. Новый проект полностью заменяет текущий сайт на `chashkacoffee.ru`.

Дизайн в это ТЗ не входит и будет описан отдельно. Архитектура, модели данных, API и админка не должны зависеть от конкретной визуальной концепции.

## 2. Цели

1. Увеличивать установки мобильного приложения.
2. Увеличивать заказы доставки и самовывоза через приложение.
3. Создать единую цифровую экосистему бренда.
4. Объединить рестораны, меню, лояльность, кофе, кондитерскую, банкеты, события, франшизу, вакансии и медиа.
5. Дать внутренней команде самостоятельную админку.
6. Собирать подробную аналитику.
7. Сделать mobile-first сайт с полноценной десктопной версией.

## 3. Базовые ограничения

Публичный сайт:

- без пользовательской авторизации и личного кабинета;
- без избранного;
- без корзины и онлайн-оплаты;
- без PWA, офлайна и web-push;
- без email- и SMS-рассылок;
- только на русском языке;
- без автоматической синхронизации с приложением;
- без deep links;
- без глобального поиска;
- без пользовательских отзывов и UGC;
- без встроенной регистрации на мероприятия.

Сайт является витриной, навигационной системой и точкой приема заявок. Заказ доставки, самовывоза, кофе, дрипов, капсул и сертификатов происходит в мобильном приложении.

## 5. Публичные разделы

### 5.1. Главная

Управляется из админки. Содержит:

- сменный промобаннер: изображение, заголовок, текст, кнопка, ссылка, даты показа, порядок;
- крупный вход в доставку;
- карточки экосистемы: рестораны, меню, приложение, лояльность, кофе, кондитерская, банкеты, события, сертификаты, франшиза, вакансии, статьи, акции;
- управляемые блоки с изменяемым порядком.

### 5.2. Рестораны

Всего 13 точек:

- 7 городских;
- 3 в парке;
- 3 в аэропорту.

Одна городская точка находится при апарт-отеле и имеет отдельный формат.

Общая страница:

- список;
- карта;
- фильтры: город, парк, аэропорт, апарт-отель.

Для каждой точки — отдельная страница:

- название, slug, формат;
- адрес;
- один телефон;
- основной график;
- временные изменения графика;
- описание;
- фотогалерея;
- превью;
- координаты;
- ссылки на Яндекс Карты и 2ГИС;
- связанное меню;
- SEO и Open Graph.

Не выводить отзывы, рейтинги, связанные акции и события, отдельные кнопки заказа, характеристики вроде парковки и веранды.

### 5.3. Интерактивное меню

PDF не является основным форматом.

Сценарий:

- пользователь выбирает точку;
- открывается меню этой точки;
- внутри меню можно переключить точку;
- со страницы ресторана меню открывается сразу для этой точки;
- выбранная точка не запоминается.

Наборы меню:

- один общий набор для 5 городских точек;
- отдельный для еще одной городской;
- отдельный для апарт-отеля;
- 3 отдельных парковых;
- 3 отдельных аэропортовых.

Система должна поддерживать общий набор для нескольких точек и локальные переопределения цены, описания и состава.

Карточка блюда:

- название;
- категория;
- описание;
- состав;
- вес/объем;
- цена;
- КБЖУ;
- аллергены;
- фиксированные маркировки;
- фото;
- маркетинговая метка;
- порядок.

Не нужны модификаторы, остатки, статусы доступности и заказ на сайте.

Фильтры: категории, вегетарианское, острое, без лактозы и другие фиксированные признаки.

## 6. Формы

Все формы:

- сохраняются в БД;
- показывают success-сообщение;
- не отправляют email/SMS пользователю;
- дублируются в Telegram;
- проходят backend-валидацию;
- имеют базовый антиспам без CAPTCHA;
- содержат согласие на обработку данных.

### Бронирование столика

Имя, телефон, точка, дата, время, количество гостей, комментарий. Проверки доступности нет; подтверждение вручную.

### Банкет

Имя, телефон, дата, количество гостей, точка, формат, комментарий.

### Кондитерская

Имя, телефон, тип заказа, готовая/индивидуальная продукция, желаемая дата, комментарий. Без конструктора и загрузки референсов.

### Франшиза

Минимум: имя, телефон, email, город, формат, бюджет, комментарий, согласие.

### Корпоративная заявка

Имя, компания, телефон, email, тип запроса, комментарий.

- ошибка отправки заявки;
- неуспешный деплой.

## 9. Админка

### Авторизация

- логин/пароль;
- без 2FA;
- без email-восстановления;
- пароль сбрасывает суперадминистратор;
- автоматический logout после бездействия, ориентир 12 часов;
- защита от brute force;
- журнал входов/выходов.

### Роли

- суперадминистратор;
- маркетолог;
- контент-менеджер;
- управляющий точки;
- менеджер франшизы;
- HR;
- менеджер банкетов;
- менеджер кондитерской.

Поддержать права по разделам, операциям и конкретным точкам. Workflow согласования не нужен.

### Контентные модули

Главная, баннеры, рестораны, графики, меню, блюда, аллергены, маркировки, кофе, кондитерская, сертификаты, акции, события, статьи, «О компании», франшиза, вакансии, корпоративный раздел, контакты, legal, SEO, пользователи, роли, заявки и настройки.

### Публикация

- сразу;
- по расписанию;
- автоматическое снятие;
- без черновиков.

### История изменений

Хранить пользователя, дату, поля, старые и новые значения. Откат не нужен.

### Аудит

Логировать входы, пароли, роли, CRUD, публикации, удаление, статусы заявок, ответственных и системные операции.

### Soft delete

- объект скрывается, но остается в БД;
- восстановления в интерфейсе нет;
- через 90 дней физическое удаление;
- журнал сохраняется согласно технической политике.

## 16. Cookie consent

- согласие/отказ от необязательной аналитики;
- ссылка на политику;
- сохранение выбора;
- возможность изменить позже;
- Метрика не активируется до допустимого согласия.

## 17. Безопасность

- backend validation и sanitization;
- XSS/CSRF защита по архитектуре;
- безопасные cookie;
- Argon2 или аналог;
- rate limiting;
- brute-force protection;
- MIME-проверка;
- разумные серверные лимиты файлов;
- honeypot;
- журнал подозрительных запросов;
- секреты только в env/secrets;
- защищенные админские маршруты.

CAPTCHA не использовать.

## 18. Персональные данные

Заявки и резюме хранятся бессрочно до ручного удаления.

Хранить:

- дату согласия;
- версию текста согласия;
- ограничивать доступ ролями.

После soft delete физическое удаление через 90 дней.

- контейнеризация;

- PostgreSQL;
- persistent media volume;
- local, staging, production;
- env;
- миграции;
- health checks;
- scheduler/cron;
- логи;
- server-level backups.

Внутреннюю систему резервного копирования не делать.

- lint;
- tests;
- build;

- migrations;
