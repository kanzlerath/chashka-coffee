# Рестораны и меню: первый продуктовый срез

Этот документ фиксирует модель для первого работающего маршрута сайта: каталог ресторанов → страница точки → меню этой точки. Он описывает текущий фундамент, а не заменяет полное ТЗ.

## Рестораны

У каждой точки есть формат и независимая географическая классификация.

- `format`: городская точка, парк, аэропорт или апарт-отель — то, как точка представлена гостю.
- `area`: город, парк или аэропорт — для основного фильтра каталога.
- `isAtApartHotel`: отдельный фильтр. Благодаря ему точка апарт-отеля одновременно остаётся городской и попадает в фильтр «Апарт-отель».

Базовая карточка публично показывает только то, что требуется для выбора точки: название, адрес, телефон, часы и обложку. Характеристики вроде Wi‑Fi, парковки, веранды, рейтингов и отзывов в эту модель намеренно не входят.

Основной график хранится построчно по дням недели; временные изменения — отдельными датированными исключениями. Это позволяет не переписывать обычное расписание ради праздника или сокращённого дня.

## Меню

`Menu` — это набор меню, а не копия меню каждой точки. Несколько ресторанов могут быть назначены одному набору через `RestaurantMenu`.

```text
Restaurant ──< RestaurantMenu >── Menu ──< MenuCategory ──< MenuItem
      │                                                   │
      └──────────────< MenuItemOverride >─────────────────┘
```

`MenuItemOverride` хранит только локальные отличия точки: цену, описание, состав и вес. Общий набор блюд не дублируется; это покрывает общий набор для пяти городских точек и любые дальнейшие локальные отличия.

Цены хранятся целым числом копеек (`priceKopecks`), чтобы не получать ошибки округления. КБЖУ хранится в явных полях, аллергены — в нормализованном справочнике, фиксированные пищевые признаки — отдельными boolean-полями. В публичном API они преобразуются в ограниченный список маркеров: вегетарианское, острое, без лактозы, без глютена, лёгкое.

## Публичный контракт

`packages/contracts/src/catalog.ts` — единственный источник типов для первой публичной витрины. Он не содержит внутренних полей: статусов редактирования, прав сотрудников, системных ключей или медиатехнических деталей.

Локальный read-only API использует эти же DTO:

- `GET /api/restaurants?area=CITY&apartHotel=false` — каталог точек с фильтрами;
- `GET /api/restaurants/:slug/menu` — назначенное точке меню с применёнными локальными переопределениями.

## Визуальная рамка, пока дизайн не финализирован

Временный ориентир для будущих экранов: светлая, тактильная и спокойная среда, в которой фотография реальной точки или блюда несёт основную эмоцию, а навигация и информация остаются почти невидимыми. Первый экран каждого раздела должен отвечать на один вопрос — «где мы?» или «что выбрать?» — без мозаики мелких карточек. Это направление основано на переданных макетах и будет уточнено отдельным дизайн-документом; цвета, типографика и анимации здесь не утверждаются.
